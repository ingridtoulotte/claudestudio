# ClaudeStudio v0.5.1 — Maintainer's Analysis

This document is the reasoning behind the v0.5.1 sprint: who hurts, what we
shipped for them, and why these particular features make ClaudeStudio the
reference implementation for local Claude Code tooling. Written from the seat of
a maintainer with thousands of daily users.

---

## 1. Top pain points in v0.5.0 — and how v0.5.1 answers each

1. **"My index is always out of date."**
   *As a user who runs Claude Code all day, I want my history to be current
   without remembering to hit Sync.* — v0.5.0 required a manual `index`/Sync.
   **Fixed by:** the **hook integration** (`claudestudio hook install` wires
   `claudestudio index` to Claude Code's `SessionEnd`) plus **live watch + SSE**
   (`claudestudio watch` and the in-app "new sessions available" toast). The loop
   closes itself.

2. **"I found the right session, but not the right *moment*."**
   *As a user with 300 sessions, I want to flag the exact message where the fix
   landed and jump back to it later.* — You could favorite a whole session, never
   a message. **Fixed by:** **message bookmarks** with a global Bookmarks view and
   deep links to `session/{id}?seq={n}`.

3. **"The replay shows me the tool call, not what actually changed."**
   *As a reviewer, I want to see the diff of an edit, not raw `old_string` /
   `new_string` blobs.* **Fixed by:** **inline unified diffs** computed with
   `difflib`, rendered with add/del coloring and a Diff/Raw toggle.

4. **"I can't share what I did this week."**
   *As someone in a standup, I want a clean summary I can paste or attach.*
   **Fixed by:** the **activity report** — a single self-contained, print-ready
   HTML (or Markdown) file for any date range.

5. **"I have Claude Code on two machines."**
   *As a power user, I want one index across my laptop and my workstation.*
   **Fixed by:** **multi-root support** — `--root a:b`, a `?root=` filter, and
   per-root counts in `doctor`/`info`.

Secondary friction also addressed: no `--version` (embarrassing), no per-tool
*latency* (only counts), and no way to see the prompts you keep retyping
(**prompt patterns**).

## 2. Competitive positioning

- **vs. Claudia (the Tauri GUI):** Claudia is a beautiful native app, but it is a
  compiled binary with a Rust/Tauri toolchain and a heavier install. ClaudeStudio
  is **`pipx run` away**, **zero runtime dependencies**, pure Python stdlib +
  vanilla JS, and ships an **MCP server** so Claude Code can query your history
  back. Different bet: *infinitely hackable and trivially auditable* over *native
  chrome*.
- **vs. raw `jq` / shell scripting:** `jq` over `~/.claude/projects/**/*.jsonl`
  works until you want FTS5 search, cost accounting, replay, analytics, or
  grounded Q&A. ClaudeStudio is the batteries-included version, and its
  `parse_session` is a documented public API so you can still script on top.
- **vs. generic log viewers:** they don't understand the Claude Code wire format,
  token/cost model, tool taxonomy, or session structure. Every feature here is
  *Claude-Code-native*.

The throughline: **local-first, zero-dependency, auditable**. Your prompts and
code never leave the machine; the server binds to `127.0.0.1`; `ask` and the MCP
tools are *computed*, not generated by a model.

## 4. Architecture decisions (and why the constraints made them better)

- **SSE over WebSockets.** Server-Sent Events are one-directional and need no
  framing library — `http.server` can stream them with a `text/event-stream`
  content type and a flush loop. WebSockets would have meant a handshake and a
  dependency. SSE keeps the **zero-dependency** promise and is exactly the right
  shape (server→client notifications).
- **Unified diff over AST diff.** `difflib.unified_diff` is stdlib, language
  agnostic, and deterministic. A semantic/AST diff would need per-language parsers
  (a dependency tree and a maintenance burden) for marginal gain in a log viewer.
- **Polling over inotify.** File-watch APIs are per-OS (`inotify`/`FSEvents`/
  `ReadDirectoryChangesW`) and flaky to abstract. A 5-second mtime poll is
  trivially **cross-platform** and cheap because `claudestudio index` is
  incremental.
- **In-place schema migration (v1→v2).** Adding multi-root meant a new `sources`
  column. Rather than rebuild, `maybe_migrate` adds it idempotently and bumps the
  recorded version, preserving all indexed data and user state — and still fails
  loudly if a *newer* index is opened by an older build.
- **User-state tables never wiped.** Bookmarks (like favorites/notes/saved
  searches) live in their own table so a reindex never touches them.

## 5. Star-growth mechanics

The features that travel on their own:

- **The report HTML.** A self-contained, good-looking file people paste into
  standups, retros, and tweets — every share carries a ClaudeStudio footer link.
- **Bookmarks deep links.** A URL that jumps to an exact message is inherently
  shareable with a teammate ("look at message 42 of this session").
- **Prompt patterns.** "ClaudeStudio told me I've asked it to *write tests for X*
  47 times" is a screenshot people post.
- **The hook moment.** `claudestudio hook install` → "wait, it indexes itself
  now?" is the kind of small delight that earns a star to save it for later.

Each of these is organic discovery: the artifact is the advertisement, and it
stays true to the local-first ethos — nothing is uploaded, the user chooses what
to share.
